# 汽车全生命周期内容统一设计

## 背景与目标

当前平台主体已经采用汽车数据跨境场景，但法规覆盖范围、服务机构、培训课程、安全事件和历史评估中仍混有医药、医疗、互联网、金融、零售、民航等独立行业案例。这些内容会削弱演示叙事的一致性，也会让跨模块关联显得像多套样例的拼接。

本次重构在不改变页面结构、路由和主要交互的前提下，将所有页面中的行业示例与演示数据统一为汽车全生命周期主题。完成后，平台从法规智库到报告中心应围绕同一套汽车业务语境展开，不再出现医药、医疗及其他非汽车行业的独立案例。

## 内容边界

平台示例统一使用以下六类业务场景：

1. 整车研发设计：车型平台参数、工程图纸、仿真模型、研发测试日志和跨国研发协同。
2. 道路测试与智能驾驶：道路测试视频、点云、车辆轨迹、环境感知和算法训练数据。
3. 生产制造与供应链：生产计划、工艺参数、零部件批次、质量追溯和供应商协同。
4. 车联网运行：车辆运行状态、服务调用日志、安全告警、联网统计和境外运营支持。
5. OTA 软件升级：软件版本、升级包、签名校验、漏洞处置、升级日志和境外开发运维。
6. 售后诊断与质量改进：车辆故障码、远程诊断日志、维修反馈、缺陷分析和质量闭环。

“远程诊断”是汽车售后与软件运维术语，“节点健康”是系统运行状态术语，二者不属于医疗示例，继续保留。

## 模块内容设计

### 法规智库

- 法规行业标签、覆盖范围、说明文字和推荐问题仅使用汽车、工业制造、车联网、智能驾驶、研发设计、软件升级及供应链相关语义。
- 删除医药、医疗器械、互联网、金融、零售和民航等独立行业示例。
- 法规名称和机制说明保持当前演示性质，不引入新的真实法规结论。

### 合规研判

- 所有场景、历史评估和结果记录映射到六类汽车业务场景。
- 将含有医疗语义的显示名称与内部标识同步改名，所有引用点保持一致。
- 路径结论继续围绕重要数据识别、个人信息规模、接收方能力与申报路径展开。

### 数据治理与评估

- 保持现有汽车资产体系，将字段、用途、数据主体、目标地区和责任部门统一到汽车业务语境。
- 不改变分类分级、重要数据识别和风险矩阵的交互逻辑。
- 检查通用输入提示和说明，确保示例用途不再出现其他行业场景。

### 申报备案

- 材料名称、项目名称、历史记录和补件提示使用汽车研发、道路测试、车联网、OTA 和供应链项目。
- 保持安全评估、标准合同与认证三类演示路径及其现有流程。

### 安全流通

- 安全事件改为研发平台权限变更、升级包异常下载、跨境研发节点访问、车联网链路异常和供应商协同权限等案例。
- 安全方案、传输监测和审计记录继续使用系统健康、节点状态等技术语义。

### 服务中心

- 服务机构名称、行业标签、能力标签和简介统一为汽车数据治理、智能驾驶、工业安全、OTA 安全和供应链协同。
- 培训课程、咨询记录和服务申请统一使用汽车全生命周期案例。

### 报告中心、首页与搜索

- 报告主题覆盖六类汽车业务场景，并与资产、评估和法规标签形成可理解的关联。
- 首页统计、动态、快捷入口和流程说明使用统一术语。
- 全局搜索结果不得因旧 mock 数据继续暴露非汽车行业案例。

## 实现策略

采用“场景体系重构”而非简单关键词替换：

- 先盘点 `src/mock` 与各模块 `data.ts` 中的内容源，再处理页面内硬编码文案。
- 保持现有 TypeScript 类型和页面组件接口，除非内部标识同步改名确有必要。
- 对含非汽车语义的记录进行完整重写，确保名称、摘要、标签、结论和引用关系一致。
- 对被其他模块引用的 ID 先查清所有引用，再一次性同步更名，避免搜索、详情页或历史记录断链。
- 不进行与内容统一无关的组件重构或视觉改版。

## 验收标准

1. 用户可见内容中不出现医疗、医药、医院、患者、病例、临床、药品、医疗器械及对应英文变体。
2. 用户可见的行业示例不出现互联网、金融、零售、民航等独立行业案例；法规通用制度名称中的必要通用词不视为行业示例。
3. 所有示例数据均能归入六类汽车全生命周期场景之一。
4. “远程诊断”和“节点健康”仅以汽车或系统运维含义出现。
5. 被改名的内部 ID 不存在旧引用或断链。
6. TypeScript Web 与 Electron 类型检查通过。
7. Vite Web 构建与 Electron TypeScript 构建通过。
8. 对首页、法规智库、合规研判、数据治理、安全流通、服务中心、报告中心和全局搜索进行关键内容抽查。

## 非目标

- 不调整页面信息架构、路由层级或导航名称。
- 不新增后端接口、数据库或远程数据源。
- 不重做视觉样式和交互流程。
- 不对法规内容作法律有效性更新或真实性背书。
